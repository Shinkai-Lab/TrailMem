# TrailMem — プライバシーと外部送信について / Privacy & What Leaves Your Machine

このドキュメントは実装（`daemon/cli.py` / `daemon/ingest.py` / `daemon/config.py` / `daemon/daemon.py`、および `trailmem-*.sh`）を根拠に書いています。断定できない挙動は「〜と思われる」ではなく実際のコードの分岐を示します。

This document is written directly from the implementation (`daemon/cli.py`,
`daemon/ingest.py`, `daemon/config.py`, `daemon/daemon.py`, and the
`trailmem-*.sh` scripts). Nothing here is speculative — every claim maps to a
specific code path.

---

## データフロー / Data Flow

```
会話ログ(jsonl)                daemon (ローカルプロセス)              LLMバックエンド              SQLite (ローカル)
~/.claude/projects/**/*.jsonl  --読み取り(watchdog監視)-->  ingest_chunk()  --記銘プロンプト-->  claude -p / Anthropic API /   --記憶を書き込み-->  ~/.trailmem/trailmem.db
(元の場所から動かない/コピーしない)                          (daemon/ingest.py)                    OpenAI互換エンドポイント

                                想起(recall/spread/flashback)は SQLite に対する
                                純粋なクエリのみ。LLM呼び出しは一切なし。
```

```
Conversation logs (jsonl)      daemon (local process)                LLM backend                    SQLite (local)
~/.claude/projects/**/*.jsonl  --read (watchdog) -->  ingest_chunk()  --encoding prompt-->  claude -p / Anthropic API /   --write memories-->  ~/.trailmem/trailmem.db
(never moved or copied)                                (daemon/ingest.py)                    OpenAI-compatible endpoint

                                Recall (recall/spread/flashback) is pure SQLite
                                query + graph math. No LLM call is ever made for it.
```

要点 / Key points:
- 記銘（エピソード抽出）だけがLLMを呼びます。想起は完全にLLMレス（純SQLite、`trailmem-scan.sh` / `trailmem-recall.sh` / `trailmem-spread.sh`）。
- Only *encoding* (episode extraction) calls an LLM. *Recall* is 100% LLM-free (pure SQLite queries in `trailmem-scan.sh` / `trailmem-recall.sh` / `trailmem-spread.sh`).

## ローカルに残るもの / What stays on your machine

すべて既定では `~/.trailmem/`（`TRAILMEM_HOME` 環境変数で変更可、`daemon/config.py`）配下に置かれます。

All of the following live under `~/.trailmem/` by default (overridable via the
`TRAILMEM_HOME` env var — see `daemon/config.py`):

| ファイル / File | 内容 / Contents |
|---|---|
| `trailmem.db` | SQLite本体。episodes（要約・感情値・source_ref）、keywords、episode_keywords（強度・想起履歴）、keyword_edges（連想網）、trailmem_meta。`daemon/ingest.py` の `SCHEMA_SQL` 参照 |
| `config.json` | daemon設定。**`anthropic_api_key` / `openai_api_key` を設定した場合、平文でここに保存されます**（`daemon/config.py` `save_config`） |
| `state.json` | 各jsonlの読み取りオフセット、処理済みUUID、最終ingest時刻（`daemon/state.py`） |
| `daemon.log` | daemonのログ |
| `control.json` | pause/resume フラグ |
| `daemon.pid` | 実行中PID |
| `flashback-buffer.jsonl` | 発火したフラッシュバックの一時バッファ（DBと同じディレクトリ。次回ingest時にLLMへ相乗り判定させた後、消費されて消える） |

**元の会話ログ（`~/.claude/projects/**/*.jsonl` など）はTrailMemによって一切コピー・移動されません。** daemonは行を読むだけで、書き込みも削除も行いません（`daemon/daemon.py` `_read_new_lines` は `open(fpath)` で読むのみ）。

**Your raw conversation logs (e.g. `~/.claude/projects/**/*.jsonl`) are never
copied or moved by TrailMem.** The daemon only reads lines from them
(`daemon/daemon.py`'s `_read_new_lines` opens the file read-only); it never
writes to or deletes the source jsonl.

埋め込み（`sqlite_vec` + `sentence-transformers`）を使う構成の場合、埋め込みベクトルもローカルの `episode_vec` テーブルに保存されます。生成もローカルのモデル（`all-MiniLM-L6-v2`）で行われ、外部には送られません（`daemon/ingest.py` `_load_embedder`）。この依存が入っていない環境では黙ってスキップされ、DBは埋め込みなしで書かれます。

If the optional `sqlite_vec` + `sentence-transformers` stack is installed,
embedding vectors are also stored locally in the `episode_vec` table. They're
generated by a local model (`all-MiniLM-L6-v2`) and never sent anywhere
(`daemon/ingest.py`'s `_load_embedder`). Where that dependency isn't
installed, embedding is silently skipped — episodes are still written, minus
the vector.

## 外部に送られるもの / What leaves your machine

記銘（episode抽出）が走るたびに、以下がLLMバックエンドに送信されます（`daemon/ingest.py` `ingest_chunk` → `call_llm`）:

Every time encoding (episode extraction) runs, the following is sent to your
configured LLM backend (`daemon/ingest.py`'s `ingest_chunk` → `call_llm`):

1. **直近の会話テキスト**: そのチャンクに含まれる user/assistant メッセージ、末尾100件まで、各メッセージは1200文字に切り詰め（`parse_jsonl_messages` / `build_conversation`）
   **Recent conversation text**: up to the last 100 user/assistant messages in
   the chunk, each truncated to 1200 characters (`parse_jsonl_messages` /
   `build_conversation`).
2. **既存キーワード一覧**: DB内のキーワードを最大200件（`existing_keywords`。「新しいキーワードを作るより既存を再利用してほしい」という指示のため）
   **Existing keyword list**: up to 200 keywords already in the DB
   (`existing_keywords` — sent so the LLM reuses existing keywords instead of
   minting near-duplicates).
3. **感情値キャリブレーション用の過去エピソード抜粋**: ランダムに3件、要約の先頭60文字＋感情値（`calibration_samples`）
   **A few past-episode excerpts for emotion calibration**: 3 random rows,
   first 60 chars of the summary + sentiment values (`calibration_samples`).
4. **保留中のフラッシュバック一覧**: `flashback-buffer.jsonl` にある、直前に発火したフラッシュバックのepisode_id・turn・要約（最大 `judge_max_items` 件、既定30件）。「used/ignored/misleading」を判定させるため（`FLASHBACK_JUDGE_SECTION`）
   **Pending flashback list**: the episode_id/turn/summary of flashbacks that
   fired since the last ingest, read from `flashback-buffer.jsonl` (up to
   `judge_max_items`, default 30) — sent so the LLM can judge
   used/ignored/misleading (`FLASHBACK_JUDGE_SECTION`).
5. **persona文字列**: config.jsonに設定していれば、その一人称ペルソナ文もプロンプト冒頭に含まれます
   **The persona string**, if you set one in config.json — it's prepended to
   every encoding prompt verbatim.

送信先はconfigの `llm_backend` で選んだ先だけです。3方式のいずれか1つにしか同時に送られません:
This goes to whichever single backend `llm_backend` selects — never more than one at a time:

- `"cli"`（既定）: ローカルの `claude` コマンドをサブプロセスで呼びます（`subprocess.run(["claude", "--model", model, "-p", ...], input=prompt)`）。これはローカル実行のCLIですが、その裏でユーザーのClaude認証を使ってAnthropicのサーバーと通信します。
  Runs the local `claude` binary as a subprocess (`_call_cli`). It's a local
  process, but under the hood it talks to Anthropic's servers using your
  existing Claude Code authentication.
- `"api"`: Anthropic APIへ直接HTTPSリクエスト（`anthropic` パッケージ経由、`_call_anthropic`）
  Direct HTTPS request to the Anthropic API via the `anthropic` package
  (`_call_anthropic`).
- `"openai"`: `openai_base_url`（既定 `https://api.openai.com/v1`）へ直接HTTPSリクエスト（`urllib.request`、`_call_openai`）
  Direct HTTPS request to `openai_base_url` (default
  `https://api.openai.com/v1`) via `urllib.request` (`_call_openai`).

想起（recall/spread/flashback判定そのもの）は上記のどれも呼びません。門1/門2のスコア計算・弱い子優先ソートはすべてローカルSQLiteのクエリと算術だけです。

Recall itself — the gate-1/gate-2 scoring, weakest-first sort, flashback
firing — never calls any of the above. It's local SQLite queries and
arithmetic only.

**送信後のデータの扱いはTrailMemの管理外です。** LLMバックエンド（Anthropic / OpenAI互換サービス）側の保持・学習利用ポリシーは各社の規約に従います。TrailMemはリクエストを送るだけで、応答（JSON）を受け取ったらそのままSQLiteに書き込むのみで、送信履歴を別途保持しません。

**What happens to the data after it leaves your machine is outside TrailMem's
control.** Retention/training-use policy on the LLM backend side (Anthropic /
your OpenAI-compatible provider) follows that provider's own terms. TrailMem
itself doesn't keep a separate log of what it sent — it sends the prompt,
receives the JSON response, and writes it to SQLite.

## APIキーの置き場所 / Where API keys live

- `~/.trailmem/config.json` の `anthropic_api_key` / `openai_api_key` フィールドに**平文**で保存されます（`daemon/config.py` `save_config` は素の `json.dump`、暗号化なし）
  Stored in **plaintext** in `anthropic_api_key` / `openai_api_key` fields of
  `~/.trailmem/config.json` (`save_config` does a plain `json.dump`, no
  encryption).
- config.jsonにキーを書かなければ、環境変数 `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` にフォールバックします（`load_config`）。値はメモリ上でのみ使われ、ファイルには書き戻されません。
  If left blank in config.json, `load_config` falls back to the
  `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` environment variables. Those values
  are only used in-process and are never written back to disk.
- 既定の `llm_backend: "cli"` を使う限り、TrailMem自体はAPIキーを一切扱いません（`claude -p` に認証を委任するため）。APIキーの管理が気になる場合は `"cli"` のまま使うのが最もシンプルです。
  As long as you stick with the default `llm_backend: "cli"`, TrailMem never
  touches an API key at all — authentication is delegated entirely to
  `claude -p`. If key handling is a concern, staying on `"cli"` is the
  simplest option.

## ログ・DBの削除方法 / Deleting logs and the DB

TrailMemには専用のuninstallコマンドはありません。状態はすべて `~/.trailmem/`（または `TRAILMEM_HOME`）1ディレクトリに閉じているため、それを削除すれば完了です。

There's no dedicated uninstall command. All state lives under the single
`~/.trailmem/` directory (or `TRAILMEM_HOME`), so removing it is sufficient.

```bash
# daemonを止めてから削除
python -m daemon.cli stop
rm -rf ~/.trailmem
```

- これで `trailmem.db`（記憶そのもの）、`config.json`（APIキーを含む）、`state.json`、ログ、バッファがすべて消えます。
  This removes `trailmem.db` (the memory itself), `config.json` (including any
  stored API key), `state.json`, logs, and buffers — all of it.
- **元の会話ログ（`~/.claude/projects/` 配下など）はこの操作の影響を受けません。** TrailMemが触るのは読み取りだけなので、`~/.trailmem` を消しても元の会話ログはそのまま残ります（逆に、元の会話ログを消してもTrailMem側の記憶は残ります — 両者は独立です）。
  **Your original conversation logs (e.g. under `~/.claude/projects/`) are
  unaffected.** TrailMem only ever reads them, so deleting `~/.trailmem`
  leaves them untouched (and conversely, deleting the original logs doesn't
  touch TrailMem's memory — the two are independent).
- TrailMem自体は削除したデータの外部コピーを保持しません。ただし、記銘時にLLMへ送信済みのプロンプト（会話テキストの抜粋）がLLMバックエンド側でどう扱われるかはそのプロバイダのポリシー次第です（前セクション参照）。「TrailMemを消せば送信済みの記憶も完全に消える」という意味ではありません。
  TrailMem itself keeps no external copy of deleted data. However, prompts
  already sent to the LLM backend during past encoding runs (excerpts of
  conversation text) are subject to that provider's own retention policy (see
  the section above) — deleting TrailMem does **not** retroactively erase
  anything already sent to the LLM backend.
- 個別のエピソードだけを弱めたい／消したい場合は `trailmem-doctor.sh set-strength EPISODE_ID KEYWORD 0` で強度を落とすか、DBを直接SQLiteで編集してください。TrailMemの「忘却」は強度を0に近づける設計で、行の物理削除を伴う専用コマンドは（`trailmem-doctor.sh` の dangling リンク削除などの整合性修復を除き）ありません。
  To weaken or suppress a single episode without wiping the whole DB, use
  `trailmem-doctor.sh set-strength EPISODE_ID KEYWORD 0`, or edit the SQLite
  file directly. TrailMem's "forgetting" is strength decaying toward zero by
  design; there's no dedicated command for physically deleting a single
  episode row (aside from `trailmem-doctor.sh`'s consistency-repair pruning of
  dangling links).

## オプトイン確認 / Opt-in confirmation

`python -m daemon.cli start` は初回、確認なしにデータを送り始めることはありません。監視ディレクトリ・DB保存先・使用するLLMバックエンドを表示した上で `y/N` の確認を求めます（`-y` で明示的にスキップ可能）。`daemon/cli.py` `cmd_start` 参照。

`python -m daemon.cli start` does not silently begin sending data on first
run. It prints the watch directory, DB path, and which LLM backend will
receive conversation text, then asks for `y/N` confirmation (skippable with
`-y`). See `cmd_start` in `daemon/cli.py`.
